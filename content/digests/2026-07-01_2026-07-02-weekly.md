# The Installer Is the Control Plane

The July 2 case is not a new vendor model or a benchmark. It is an installer. At
[`v0.7.0`](https://github.com/Dicklesworthstone/agentic_coding_flywheel_setup/releases/tag/v0.7.0),
[Agent Flywheel](/profiles/agent-flywheel/) writes a VPS posture where
[Claude Code runs with `--dangerously-skip-permissions`, Codex runs with
`--dangerously-bypass-approvals-and-sandbox`, and Antigravity runs through an
ACFS wrapper](https://raw.githubusercontent.com/Dicklesworthstone/agentic_coding_flywheel_setup/v0.7.0/acfs/zsh/acfs.zshrc).
In the same tagged source, vibe mode
[writes `NOPASSWD:ALL` into sudoers](https://raw.githubusercontent.com/Dicklesworthstone/agentic_coding_flywheel_setup/v0.7.0/scripts/lib/user.sh),
and the Antigravity wrapper
[sets `always-proceed`, turns the terminal sandbox off, allows non-workspace
access, and invokes `agy` with `--dangerously-skip-permissions`](https://raw.githubusercontent.com/Dicklesworthstone/agentic_coding_flywheel_setup/v0.7.0/scripts/lib/agy_locked.py).
The cost is not hidden in the fine print. The one-command stack makes a beginner's
setup easier by moving the authority decision upstream, to the person who chose
the defaults.

That is the assembly layer, and it deserves a different read from a normal tool
release. Anthropic, OpenAI, and Google still own their harnesses. Agent Flywheel
owns the bundle a new operator may actually run. It decides that Claude should
skip permissions, Codex should bypass approvals and sandboxing, Antigravity should
always proceed, and the host account should have passwordless sudo. The project is
not coy about the target. Its README says full vibe mode is for
[throwaway VPS environments](https://raw.githubusercontent.com/Dicklesworthstone/agentic_coding_flywheel_setup/v0.7.0/README.md)
and lists safe mode as the path with no passwordless sudo and standard
confirmations. The mistake would be to treat that as a warning label after the
fact. It is the product boundary.

## The third-party default

A harness vendor can expose a dangerous flag and still leave the operator in
charge of whether to use it. An installer changes that relation. It can collapse
three separate decisions into one command pasted into a shell. ACFS does that for
three tier-1 agents and a VPS account. The operator who runs the default path is
not choosing each agent's governance posture; they are accepting the bundle's
governance posture.

The Antigravity wrapper is the clearest specimen because it is a policy file, not
copy. The tagged `agy_locked.py`
[pins settings](https://raw.githubusercontent.com/Dicklesworthstone/agentic_coding_flywheel_setup/v0.7.0/scripts/lib/agy_locked.py)
for model, tool permission, artifact review policy, telemetry, terminal sandbox,
non-workspace access, and other behavior. It also filters user-provided `--model`,
`--sandbox`, and dangerous-skip flags before calling the real binary. That is a
real control plane. It is not the operator remembering a flag. It is a wrapper
that keeps restoring policy on launch and on exit.

There is safety machinery in the same stack, but it has to be read at the same
precision. The ACFS launcher wires a Destructive Command Guard hook for
Antigravity. If `dcg` is unavailable or times out, the hook
[emits `allow` with a fail-open reason](https://raw.githubusercontent.com/Dicklesworthstone/agentic_coding_flywheel_setup/v0.7.0/scripts/lib/agy_locked.py).
That is better than no guard in normal operation. It is not a hard boundary under
guard failure. On a disposable VPS, that may be the intended trade. On a laptop,
shared host, or credential-rich server, it is the incident shape.

The economics are also part of the posture. The tagged web app source says the
setup expects an operator willing to invest
[about `$500/month` in AI subscriptions and shows an estimated `$440-656/month`
total](https://raw.githubusercontent.com/Dicklesworthstone/agentic_coding_flywheel_setup/v0.7.0/apps/web/app/page.tsx)
for VPS, Claude Max, and ChatGPT Pro. That is not a criticism. It is the right
unit of adoption: a paid personal operating cell, not a free CLI.

## The velocity receipt is narrow

The source joined the watchlist partly because the author is an extreme-use
specimen, and that claim needed a receipt. The receipt is not the author's other
repositories; those stayed outside scope. It is Robbes et al.,
["Agentic Very Much! Adoption of Coding Agent in New GitHub Projects"](https://arxiv.org/abs/2606.07448),
submitted 2026-06-05. The arXiv source bundle behind the paper renders the
newer-project top-35 organization figure with `dicklesworthstone` leading at 110
adoptions, and its table sets Microsoft's newer-project repository count to 52.
The paper prose says this single developer created more repositories with
coding-agent traces than Microsoft during the study period.

That verifies a narrower thing than internet folklore usually wants. It verifies
one paper's measure of repositories with coding-agent traces during its study
period. It does not verify every self-reported productivity number, it does not
license harvesting a whole portfolio, and it does not prove the installer is safe.
It explains why this source belongs on the watchlist: it is one operator's
defaults, exported as an onboarding path for other operators.

## The vendors moved around the same boundary

The rest of the July 1-2 window did not overturn the thesis. It circled it.

[Claude Code 2.1.198](https://github.com/anthropics/claude-code/releases/tag/v2.1.198)
makes subagents run in the background by default and says background agents
launched from `claude agents` now commit, push, and open draft PRs when worktree
code work finishes instead of stopping to ask. That is useful if your review
boundary is the draft PR. It is dangerous if your mental model is still "the
background agent is just thinking somewhere else." The control moved from
foreground attention to branch protection, notifications, and review policy.

[Hermes v2026.7.1](https://github.com/NousResearch/hermes-agent/releases/tag/v2026.7.1)
is the week's clean upgrade: last issue's main-only security wave reached a tag.
The release notes list MCP-config persistence hardening, cron `base_url`
credential-exfiltration blocking, prefix-secret sentinels for file reads, Slack
`xapp-` token redaction, browser cloud-metadata guardrails, resume/session
scoping, and a dependency floor. The operator action is boring and therefore
valuable: upgrade from `v2026.6.19`.

[Codex rust-v0.142.5](https://github.com/openai/codex/releases/tag/rust-v0.142.5)
closes a quieter leak: full Responses WebSocket request payloads no longer go to
trace logs. Anyone retaining or centralizing traces should upgrade and then audit
old logs as if they may contain prompts, repository context, or tool inputs. A log
is a permission boundary after the fact. It is where the thing you allowed
yourself to run gets copied for someone else to read.

Gemini's open repo fixed a sharper file boundary, but not for stable users yet.
Nightly
[`v0.51.0-nightly.20260702.gff00dacd9`](https://github.com/google-gemini/gemini-cli/releases/tag/v0.51.0-nightly.20260702.gff00dacd9)
fixes a symbolic-link directory escape in the memory import processor. The
operator consequence is simple: do not process untrusted `GEMINI.md` memory
imports on the stable line as if the fix had shipped there. It has not.

## Provider notes

**Agent Flywheel** joins as a tier-2 watch source with a seeded
[profile](/profiles/agent-flywheel/). The read is an assembly-layer read: a
third party choosing dangerous defaults for several agents, explicitly aimed at a
disposable VPS.

**Claude Code** shipped the most authority-expanding vendor change in the window:
background agents can now produce commits, pushes, and draft PRs. That is not bad
by itself. It is a shift in where review is supposed to happen.

**Hermes Agent** cleared a carry-forward cleanly. Security fixes that were
main-only last issue are now in `v2026.7.1`.

**Codex** shipped the trace-log payload scrub to stable. The 0.143 line remains
alpha-only, so the runtime/telemetry work there stays watch-only.

**Gemini CLI** fixed the memory-import symlink escape in nightly only. The
`@file` hardening carry-forward also remains nightly-only.

**Antigravity** moved in its changelog, but not in a way this run can promote:
the official changelog is pinned to commit
`2939422297ba014da627961f4e387fad0e151f47`, while public tag/release URLs for
1.0.15 and 1.0.16 did not resolve during harvest.

**OpenHands** still has the same self-hosting problem: the Authlib
[`CVE-2026-44681`](https://nvd.nist.gov/vuln/detail/CVE-2026-44681) fix remains
main-only, and there is still no OSS tag after 1.8.0. The vulnerability class is
an OIDC open redirect / incorrect authorization flow, with phishing leverage from
a trusted authorization-server origin when vulnerable grants are exposed.

**eve** shipped useful governance work: subagent depth caps, token caps, Slack
approval inputs, large approval batch splitting, generic Chat SDK channel support,
and failure propagation fixes. It remains the platform-side counterexample where
delegation and approval are first-class objects.

**Agent Zero** made model setup more explicit in `v2.2`, removed silent OAuth
model auto-population, and set Sonnet 5 / GPT-5.5 defaults. Good hygiene; still a
pin-your-models story.

**OpenClaw**, **Paperclip**, and **Flue** all moved in prerelease or main-only
channels. OpenClaw's scoped capability profiles are beta, Paperclip's operating
state work is canary, and Flue's contract rewrite is unreleased.

**heypi** and **Pi** did not change the read. heypi remains at `0.2.0-beta.0`;
Pi remains a low-governance core harness with no new release after `v0.80.3`.
